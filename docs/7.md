## Обоснование выбора стека технологий

### 1. Общие положения
При выборе стека технологий для разработки мобильного приложения (Android) для обмена учебными материалами учитывались следующие критерии:
* надёжность и распространённость технологий в экосистеме Android;
* поддержка офлайн-режима и локального хранения данных;
* безопасность хранения данных на устройстве;
* возможность масштабирования функциональности в будущем;
* наличие развитой экосистемы, документации и сообщества.

### 2. Язык программирования
**Выбор:** Java  

**Обоснование выбора:**  
* Java является официальным языком разработки под Android, обладает многолетней поддержкой;
* обеспечивает высокую производительность и стабильность нативных приложений;
* имеет богатую экосистему библиотек и инструментов;
* поддерживает объектно-ориентированный подход, что упрощает модульную архитектуру;
* позволяет легко интегрироваться с Android SDK и библиотеками Google.

Java идеально подходит для реализации бизнес-логики, работы с локальными базами данных и пользовательским интерфейсом Android.

### 3. Клиентская часть (пользовательский интерфейс)
**Выбор:** Мобильное приложение для Android (Java)  

**Обоснование выбора:**  
* высокая производительность и плавность интерфейса благодаря нативной разработке;
* полный доступ к возможностям устройства (файловая система, разрешения, уведомления);
* возможность работы без постоянного подключения к интернету (офлайн-режим);
* использование стандартных компонентов Android (Activity, Fragment, RecyclerView) упрощает разработку и поддержку.

### 4. Система управления базами данных (СУБД)
**Выбор:** Room (библиотека-обёртка над SQLite)  

**Обоснование выбора:**  
* предоставляет удобный и типобезопасный API для работы с SQLite;
* является частью Android Jetpack, официально рекомендуется Google;
* поддерживает реактивные запросы (LiveData, Flow) и проверку SQL-запросов на этапе компиляции;
* упрощает миграции и тестирование базы данных;
* хорошо интегрируется с архитектурными компонентами Android (ViewModel, LiveData).

Room оптимально подходит для локального хранения учебных материалов, информации о пользователях, рейтингов и истории действий.

### 5. Средства хранения файлов
**Выбор:** Локальная файловая система устройства (внутреннее или внешнее хранилище)  

**Обоснование выбора:**  
* позволяет хранить файлы большого размера (PDF, видео, архивы) без ограничений;
* обеспечивает быстрый доступ к материалам в офлайн-режиме;
* не требует подключения к интернету и серверных мощностей;
* использование Scoped Storage (начиная с Android 10) гарантирует безопасность данных и соблюдение политик конфиденциальности.

### 6. Инструменты тестирования
**Выбор:**  
* JUnit — модульное тестирование бизнес-логики и репозиториев;

**Обоснование выбора:**  
* JUnit позволяет выявлять ошибки на ранних этапах разработки;
* тестирование на уровне базы данных и ViewModel повышает надёжность приложения;
* возможность автоматизации тестов в среде CI/CD.

### 7. Инструменты разработки и сборки
**Выбор:**  
* Android Studio — официальная среда разработки для Android;
* Gradle — система сборки и управления зависимостями;
* Git — система контроля версий.

**Обоснование выбора:**  
* Android Studio предоставляет специализированные инструменты (эмулятор, профилировщик, редактор макетов);
* Gradle упрощает конфигурацию сборки, подключение библиотек и создание разных вариантов приложения (debug/release);
* Git обеспечивает эффективную командную работу и контроль версий кода.

### 8. Средства обеспечения безопасности
**Используемые механизмы:**  
* Разрешения Android — контроль доступа к хранилищу, камере и другим функциям;
* Шифрование данных (EncryptedSharedPreferences, EncryptedFile) — для защиты чувствительной информации;
* Хеширование паролей (если реализуется локальная аутентификация) — с использованием BCrypt или аналогичных алгоритмов.

**Обоснование выбора:**  
* встроенные механизмы Android обеспечивают базовый уровень защиты данных;
* соблюдение рекомендаций Google по безопасному хранению (Scoped Storage, шифрование) предотвращает утечки;
* локальная модель безопасности исключает риски, связанные с передачей данных по сети.
